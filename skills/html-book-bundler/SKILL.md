# HTML Book Bundler

Навык для сборки книги в формат single-file HTML. Главы хранятся в Base64 и подгружаются в изолированный `iframe` через Blob URL.

## Спецификация (OpenSpec)

### Requirement: Isolation
Каждая глава **SHALL** отображаться в отдельном `<iframe>`, чтобы стили и скрипты главы не ломали оболочку книги.

### Requirement: Portability
Результирующий файл **SHALL NOT** иметь внешних зависимостей (CDN/внешний CSS/JS).

### Requirement: Visual Fidelity
Если в исходной книге есть схемы и диаграммы, итоговая интерактивная книга **SHOULD** сохранять их смысловую структуру (узлы, связи, подписи), а не только текстовые пересказы.

## Базовый процесс

1. Подготовьте главы как валидные HTML-файлы.
2. При наличии визуалов извлеките и проанализируйте схемы из PDF (см. workflow ниже).
3. Встроите схемы в главы как `canvas + JavaScript` (или растровый ассет с JS-оверлеем) + подписи.
4. Соберите итоговую книгу скриптом `bundle.cjs`.

## Гейт качества (обязательная перепроверка)

Перед сдачей результата:
1. Откройте итоговый `*.html` в браузере и проверьте навигацию (главы, prev/next, клавиши `←/→`).
2. Откройте каждую главу отдельно и проверьте отсутствие console errors.
3. Проверьте адаптивность (desktop + mobile viewport).
4. Убедитесь, что ключевые схемы читаемы без длинного текста: есть подписи + короткая интерпретация.
5. Для интерактивных блоков проверьте минимум 1 пользовательский сценарий (клик, фильтр, слайдер).

Если есть ошибки в консоли, результат считается незавершенным.

## Сборка

```bash
node skills/html-book-bundler/scripts/bundle.cjs --input ./chapters --output my-book.html
```

Аудит готового single-file:

```bash
python skills/html-book-bundler/scripts/audit_single_file_html.py --file ./my-book.html
```

## Перенос схем и картинок из PDF

Используйте helper:

```bash
python skills/html-book-bundler/scripts/extract_pdf_visuals.py --pdf ./book.pdf --out ./_pdf_assets --pages 1-20 --dpi 180
```

Дальше:
1. Сверьтесь с `pages/page_XXX.png` и `manifest.json`.
2. Выберите ключевые схемы.
3. Перенесите их в интерактив (предпочтительно `canvas + JavaScript`).
4. Добавьте под схемой краткое объяснение «что означает схема».

Для этого проекта и похожих методологических книг:
- схемы и диаграммы по умолчанию переносить как `canvas + JavaScript` (без SVG);
- интерактив схемы должен поддерживать минимум: подсветку этапа и краткую интерпретацию этапа;
- декоративные иллюстрации запрещены, только визуалы, которые помогают принять решение/выполнить практику.

Подробно: `references/pdf-visual-workflow.md`.

## Архитектурные режимы

1. `Zero-Dependency` (по умолчанию):
   - только native web APIs (`canvas`, `Blob`, `requestAnimationFrame`, CSS).
2. `Controlled Library Injection` (опционально):
   - библиотека фиксированной версии, встроенная в single-file;
   - ленивое подключение только для нужной главы;
   - обязательный fallback/деградация.

## Паттерны визуализации для книжного домена

Для методологий и управленческих книг предпочитайте:
- процессные схемы (pipeline/спираль/циклы) с подсветкой этапов;
- карточки принципов с компактной «картинкой-смыслом» (иконка/мини-диаграмма);
- матрицы «принцип → ритуал → метрика»;
- профиль зрелости (radar/score) с явной шкалой 1-5 и суммой баллов.

Избегайте «голых» декоративных визуалов без управленческого смысла.

## Уроки (из практики на этом проекте)

1. Никогда не смешивать разные источники координат для базовой схемы и подсветки.
2. Для canvas-схем фиксировать размеры `width/height` и CSS-размеры одинаково, иначе выделения «уезжают».
3. Любая попытка «починить на глаз» без browser-check только ухудшает итог.
4. После каждого изменения интерактива обязателен прогон в браузере и проверка конкретных фаз.
5. При работе с русским текстом всегда сохранять файлы в UTF-8 и сразу проверять отсутствие mojibake.
6. Если требуется 1:1 визуальная близость, вводить измеримую проверку (скрин + метрика), а не субъективную оценку.

## Политика сжатия контента (preview-фрагмент)

Если пользователь просит ознакомительный фрагмент (например, 3 главы):
1. Не сокращайте до лозунгов; сохраняйте причинно-следственную логику главы.
2. В каждой главе включайте: `выжимка`, `практические шаги`, `риски/ошибки`.
3. Сохраняйте ключевые термины и структуру оригинала (этапы, коды шагов, названия принципов).

## Сценарии

### Scenario: Добавление новой главы
WHEN пользователь добавляет HTML-главу в папку,
AND запускает `bundle.cjs`,
THEN итоговый файл обновляет массив `B64`, заголовки и навигацию.

### Scenario: Перенос ключевой схемы из PDF
WHEN в исходной книге есть важная схема,
AND пользователь собирает интерактивный фрагмент,
THEN глава включает визуальный блок (`canvas + JavaScript`/изображение с JS-оверлеем) с сохранением структуры схемы и краткой интерпретацией.
